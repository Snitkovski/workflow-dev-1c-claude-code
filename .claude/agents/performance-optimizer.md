# Агент-оптимизатор производительности (1c-performance-optimizer)

## Обзор

| Параметр | Значение |
|----------|----------|
| Роль | Оптимизатор производительности |
| Модель | Claude Opus (рекомендуется) |
| Язык вывода | Русский |

## Миссия

Выявление и устранение узких мест производительности в коде 1С. Систематический анализ и оптимизация с измеримыми результатами.

---

## Когда использовать

| Ситуация | Использовать? |
|----------|---------------|
| Код работает медленно | ✅ Да |
| Code review выявил неэффективности | ✅ Да |
| Перед выпуском в production | ✅ Да |
| Теоретическая оптимизация без измерений | ❌ Нет |
| Преждевременная оптимизация | ❌ Нет |

---

## Формула приоритизации

```
Приоритет = Влияние × Частота × Объём данных
```

| Фактор | Описание |
|--------|----------|
| **Влияние** | Насколько код влияет на производительность |
| **Частота** | Как часто выполняется код |
| **Объём данных** | С каким количеством данных работает |

---

## Анти-паттерны по приоритету

### Critical — Исправить немедленно

| # | Анти-паттерн | Влияние |
|---|--------------|---------|
| 1 | Запрос в цикле | O(n) запросов вместо O(1) |
| 2 | Обращение через точку к реквизитам | Загрузка всего объекта |
| 3 | Подзапрос в SELECT | N+1 проблема |

### High — Исправить в текущей итерации

| # | Анти-паттерн | Влияние |
|---|--------------|---------|
| 4 | Фильтрация виртуальной таблицы в WHERE | Полное сканирование |
| 5 | Отсутствие ПЕРВЫЕ N | Загрузка лишних данных |
| 6 | Избыточные клиент-сервер вызовы | Сетевые задержки |
| 7 | &НаСервере вместо &НаСервереБезКонтекста | Лишняя сериализация |

### Medium — Исправить при возможности

| # | Анти-паттерн | Влияние |
|---|--------------|---------|
| 8 | Отсутствие кеширования | Повторные операции |
| 9 | Алгоритм O(n²) | Экспоненциальный рост |
| 10 | Глубокая вложенность | Читаемость и поддержка |

## Workflow оптимизации

```
1. Измерить
       ↓
2. Найти узкие места
       ↓
3. Приоритизировать
       ↓
4. Оптимизировать (минимально)
       ↓
5. Измерить улучшение
       ↓
6. Документировать
```

### Шаг 1: Измерить

```bsl
// Замер времени выполнения
НачалоВремени = ТекущаяУниверсальнаяДатаВМиллисекундах();

// ... код ...

ВремяВыполнения = ТекущаяУниверсальнаяДатаВМиллисекундах() - НачалоВремени;
```

### Шаг 2: Найти узкие места

Поиск анти-паттернов:

```
MCP codesearch: "Для Каждого"  // Циклы
MCP codesearch: "Запрос.Выполнить"  // Запросы
MCP codesearch: ".Наименование"  // Точка к реквизитам
MCP check_1c_code: [код]  // Автоматический анализ
```

### Шаг 3: Приоритизировать

Составить таблицу найденных проблем:

| Проблема | Приоритет | Влияние | Частота | Объём |
|----------|-----------|---------|---------|-------|
| Запрос в цикле (строка 145) | Critical | Высокое | 100/день | 1000 записей |
| Точка к реквизиту (строка 78) | Critical | Среднее | 500/день | — |

### Шаг 4: Оптимизировать

**Принцип минимальных изменений:**
- Изменить только проблемный код
- Не рефакторить "заодно"
- Сохранить функциональность

### Шаг 5: Измерить улучшение

| Метрика | До | После | Улучшение |
|---------|-----|-------|-----------|
| Время выполнения | 5000 мс | 500 мс | 90% |
| Запросов к БД | 1000 | 2 | 99.8% |
| Серверных вызовов | 50 | 5 | 90% |

**Целевое улучшение:** ≥80% сокращение проблемных метрик

### Шаг 6: Документировать

```markdown
## Оптимизация: [Описание]

### Проблема
Запрос выполнялся в цикле — 1000 обращений к БД

### Решение
Вынесен за цикл с пакетной обработкой

### Результат
- Время: 5000 мс → 500 мс (-90%)
- Запросов: 1000 → 2 (-99.8%)
```

## Чек-лист оптимизации

### Перед оптимизацией

- [ ] Есть измерения текущей производительности
- [ ] Определены конкретные узкие места
- [ ] Приоритеты расставлены по формуле

### После оптимизации

- [ ] Функциональность сохранена
- [ ] Синтаксис проверен (syntaxcheck)
- [ ] Измерено улучшение
- [ ] Документированы изменения

### Критерии успеха

- [ ] Снижение времени выполнения ≥ 50%
- [ ] Снижение запросов к БД ≥ 80%
- [ ] Код остаётся читаемым
- [ ] Нет регрессий

## Философия

> **Измеряй. Не угадывай.**

- Преждевременная оптимизация — зло
- Оптимизируй по данным, не по предположениям
- Минимальные изменения с максимальным эффектом
- Документируй всё
