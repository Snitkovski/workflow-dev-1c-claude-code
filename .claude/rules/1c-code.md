# Правила написания кода 1С

> Применяется к: `**/*.bsl`, `**/*.os`

## Конфигурация проекта (.dev.env)

**Перед началом работы с кодом** прочитать `.dev.env` из корня проекта. Если файла нет — запросить параметры у пользователя. **Угадывать значения ЗАПРЕЩЕНО.**

| Параметр | Влияние на код |
|----------|---------------|
| `{PREFIX}` | Префикс для ВСЕХ новых объектов метаданных, реквизитов, элементов форм, ролей |
| `{COMPANY}` | Используется в шаблонах комментариев модификации |
| `{DEVELOPER}` | Используется в шаблонах комментариев модификации |
| `{PLATFORM_VERSION}` | Определяет доступные возможности платформы (Асинх/Ждать с 8.3.18, ОписаниеОповещения для более ранних) |
| `{COMMENT_OPEN}` | Шаблон открывающего комментария модификации |
| `{COMMENT_CLOSE}` | Шаблон закрывающего комментария модификации |
| `{NEW_OBJECTS_IN}` | Где создавать новые объекты: `main_configuration` (по умолчанию) или `extension` |

> Для задач без генерации кода (ревью, анализ, документация) — параметры не блокируют работу.

Шаблон: [.dev.env.example](../../.dev.env.example)

## Язык и кодировка

| Параметр | Значение |
|----------|----------|
| Язык кода | Русский (кириллица) |
| Кодировка файлов | UTF-8 with BOM для .bsl |
| Перевод строк | CRLF (Windows/1C стандарт) |
| Буква "ё" | НЕ использовать в коде и метаданных |

**Типографика (в коде, комментариях, синонимах метаданных, сообщениях пользователю):**
- Тире `—` НЕ использовать — заменять дефисом `-` (проблемы кодировки в платформе)
- Кавычки для пользовательских текстов: `«...»`
- Неразрывные пробелы и невидимые Unicode-символы — запрещены

**Исключения для английского:**
- Имена веб-сервисов и их методов
- Параметры HTTP-запросов
- Поля JSON-структур для внешних API

## Форматирование

### Отступы и ширина

| Параметр | Значение |
|----------|----------|
| Отступ | Табуляция (НЕ пробелы) |
| Размер табуляции | 4 символа |
| Максимальная длина строки | 120 символов |
| Один оператор на строку | Обязательно |

### Метрики качества

| Метрика | Лимит | Строгость |
|---------|-------|-----------|
| Длина метода | ≤ 200 строк (исключение: тексты запросов) | Жёсткий лимит |
| Длина метода | > 100 строк — кандидат на декомпозицию | Ревью-триггер |
| Вложенность управляющих конструкций | < 5 уровней | Жёсткий лимит |
| Когнитивная сложность | < 15 | Ревью-триггер |
| Параметры метода | ≤ 5 (дополнительные через Структуру) | Жёсткий лимит |

### Выравнивание

- Для групп однотипных присваиваний **в локальные переменные** — выравнивать `=` пробелами
- **НЕ выравнивать** при установке свойств объекта через точку — один пробел вокруг `=`

### Удаление мёртвого кода

**ОБЯЗАТЕЛЬНО удалять:**
- Неиспользуемые процедуры и функции
- Закомментированный код
- TODO без ссылки на задачу в трекере
- Пустые обработчики событий
- Декоративные баннеры из `///` и заголовки модулей с авторством/историей (это задача git)

## Именование

### Переменные (PascalCase)

**Имена ДОЛЖНЫ отражать бизнес-смысл и роль.** Суффиксы типов допустимы только когда снимают неоднозначность и не превращаются в венгерскую нотацию.

```bsl
// ПРАВИЛЬНО — бизнес-имена
Контрагенты
Заказы
ТекущийЭлемент
НоваяСтрока
СоответствиеВидовИмен

// НЕПРАВИЛЬНО — сокращения
масРекв
соотвВидИмя
новСтр

// НЕПРАВИЛЬНО — венгерская нотация
МассивКонтрагентов  // → Контрагенты
ТаблицаДанных       // → Данные
СписокДокументов    // → Документы

// НЕПРАВИЛЬНО — подчёркивание в начале
_ИмяПеременной

// ЗАПРЕЩЕНО — имена из глобального контекста 1С как локальные переменные
Документы            // конфликт с глобальным контекстом
Справочники          // конфликт
Пользователи         // конфликт
Регистры             // конфликт
Метаданные           // конфликт
Константы            // конфликт
```

### Булевы переменные

Имя должно отражать состояние `Истина`. **Только положительные имена:**

```bsl
// ПРАВИЛЬНО
ЕстьОшибки
ЭтоНовый
ДокументПроведен
РазрешеноРедактирование
ПроверкаПройдена

// НЕПРАВИЛЬНО — отрицание в имени
НетОшибок           // → ЕстьОшибки
НеПроведен          // → Проведен
ПроверкаНеПройдена  // → ПроверкаПройдена
```

### Процедуры и функции

| Назначение | Паттерн | Пример |
|------------|---------|--------|
| Возвращает описание | Существительное | `ПолноеИмя()` |
| Создаёт новый объект | Префикс "Новый" | `НовоеПолеФормы()` |
| Проверяет условие | "Это" или причастие | `ЭтоВнешняяЗадача()`, `ДокументПроведен()` |
| Выполняет действие | Глагол в инфинитиве | `ВыбратьДанныеПоПравилу()` |
| Получает данные | Префикс "Получить" | `ПолучитьДанныеЗаказа()` |
| Устанавливает значение | Префикс "Установить" | `УстановитьСтатус()` |
| Конструктор параметров | Префикс "Параметры" | `ПараметрыЗаполнения()` |

**Префикс `Получить*`** нежелателен, когда возвращаемое значение очевидно из имени или функция возвращает коллекцию (`Контрагенты()` вместо `ПолучитьКонтрагентов()`). Допустим при делегировании к вызову платформы или реализации контракта БСП.

**Описание `Экспорт` процедур/функций** — обязательно, начинается с глагола: "Возвращает...", "Проверяет...", "Рассчитывает...". НЕ начинать с "Процедура...", "Функция..." или имени функции.

### Префикс расширения

**Все экспортные функции расширения** используют префикс проекта:

```bsl
// Примеры префиксов по проектам:
// бэкс - business export
// бтех - BisOnTech
// инт  - integration

// Экспортная функция — с префиксом
Функция бэксПолучитьДанныеЗаказа(ИдентификаторЗаказа) Экспорт
    // ...
КонецФункции

// Внутренняя функция — без префикса
Функция ПреобразоватьВJSON(Данные)
    // ...
КонецФункции
```

**Определение префикса проекта:**
- Указывается в `.dev.env` (`{PREFIX}`) или `CLAUDE.md` проекта
- Используется для всех модулей расширения
- Обеспечивает уникальность имён

### Именование метаданных

| Элемент | Правило |
|---------|---------|
| Новые объекты метаданных | Префикс `{PREFIX}` в имени |
| Синонимы объектов | Без префикса. При конфликтах — `({COMPANY})` |
| Новые роли | Префикс `{PREFIX}` |
| Подсистемы | `{PREFIX}ДобавленныеОбъекты` и `{PREFIX}ИзмененныеОбъекты` |
| Реквизиты типовых объектов | Префикс `{PREFIX}` |
| Элементы типовых форм | Префикс `{PREFIX}` |
| Реквизиты внутри нетиповых объектов | **БЕЗ префикса** (имя объекта уже содержит `{PREFIX}`) |

### Строки и конкатенация

Использовать `СтрШаблон()` для составления строк, **НЕ конкатенацию через `+`:**

```bsl
// ПРАВИЛЬНО
Сообщение = СтрШаблон("Заказ %1 от %2 обработан", НомерЗаказа, ДатаЗаказа);

// НЕПРАВИЛЬНО
Сообщение = "Заказ " + НомерЗаказа + " от " + ДатаЗаказа + " обработан";
```

Исключение: простая конкатенация `Префикс + Суффикс` допустима, когда читается лучше.

### Магические числа

**ЗАПРЕЩЕНЫ.** Выносить в именованные переменные:

```bsl
// НЕПРАВИЛЬНО
Если ДнейОтгрузки > 30 Тогда

// ПРАВИЛЬНО
МаксимальныйСрокОтгрузкиДней = 30;
Если ДнейОтгрузки > МаксимальныйСрокОтгрузкиДней Тогда
```

### Перечисления строковых значений

Располагать **в алфавитном порядке.**

## Контекст выполнения (директивы компиляции)

| Операция | Директива | Причина |
|----------|-----------|---------|
| Массовые операции с данными | `&НаСервереБезКонтекста` | Производительность |
| Чтение данных для отображения | `&НаСервереБезКонтекста` | Минимизация передачи |
| Изменение данных формы | `&НаСервере` | Нужен контекст формы |
| Интерактивные действия | `&НаКлиенте` | Отзывчивость UI |
| Асинхронные операции | `&НаКлиенте` с `Асинх` | Неблокирующий UI |

### Правило предпочтения

```
&НаСервереБезКонтекста > &НаСервере
```

Используйте `&НаСервере` только когда действительно нужен контекст формы.

## Обработка ошибок

### Принципы

1. **Избегать** `Попытка...Исключение` для стандартных операций с БД
2. **Использовать** транзакции с предварительной проверкой условий
3. **Логировать** только значимые ошибки
4. **Не подавлять** исключения без обработки

### Паттерн для API / интеграций

```bsl
Функция ВыполнитьОперацию(Параметры)

    Результат = Новый Структура("Успех, Данные, Ошибка", Ложь, Неопределено, "");

    // 1. Валидация БЕЗ исключений
    ОшибкаВалидации = ПроверитьПараметры(Параметры);
    Если ЗначениеЗаполнено(ОшибкаВалидации) Тогда
        Результат.Ошибка = ОшибкаВалидации;
        Возврат Результат;
    КонецЕсли;

    // 2. Бизнес-логика с транзакцией
    НачатьТранзакцию();
    Попытка

        // Операции с данными
        // ...

        ЗафиксироватьТранзакцию();
        Результат.Успех = Истина;
        Результат.Данные = ПолученныеДанные;

    Исключение
        ОтменитьТранзакцию();

        ИнформацияОбОшибке = ИнформацияОбОшибке();
        Результат.Ошибка = КраткоеПредставлениеОшибки(ИнформацияОбОшибке);

        // Логирование для анализа
        ЗаписьЖурналаРегистрации(
            "ИмяМодуля.ВыполнитьОперацию",
            УровеньЖурналаРегистрации.Ошибка,
            ,
            ,
            ПодробноеПредставлениеОшибки(ИнформацияОбОшибке));

    КонецПопытки;

    Возврат Результат;

КонецФункции
```

### Паттерн для транзакций

```bsl
// ПРАВИЛЬНО - проверка перед действием
Если Документ.Проведен Тогда
    Возврат; // или сообщение об ошибке
КонецЕсли;

НачатьТранзакцию();
Попытка
    Документ.Записать(РежимЗаписиДокумента.Проведение);
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;

// НЕПРАВИЛЬНО - исключение для контроля потока
Попытка
    Документ.Записать(РежимЗаписиДокумента.Проведение);
Исключение
    // Подавление ошибки
КонецПопытки;
```

## Работа с данными

### Получение реквизитов ссылок

```bsl
// НЕПРАВИЛЬНО - обращение через точку
ИмяКонтрагента = Контрагент.Наименование;

// ПРАВИЛЬНО - через функцию БСП
ИмяКонтрагента = ОбщегоНазначения.ЗначениеРеквизитаОбъекта(
    Контрагент,
    "Наименование");

// ПРАВИЛЬНО - получение нескольких реквизитов
РеквизитыКонтрагента = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(
    Контрагент,
    "Наименование, ИНН, КПП");
```

### Работа с запросами

```bsl
// КРИТИЧНО: НЕ выполнять запросы в циклах!

// НЕПРАВИЛЬНО
Для Каждого Элемент Из Коллекция Цикл
    Запрос = Новый Запрос;
    Запрос.Текст = "ВЫБРАТЬ ... ГДЕ Ссылка = &Ссылка";
    Запрос.УстановитьПараметр("Ссылка", Элемент.Ссылка);
    Результат = Запрос.Выполнить(); // N запросов!
КонецЦикла;

// ПРАВИЛЬНО - пакетный запрос
Запрос = Новый Запрос;
Запрос.Текст = "ВЫБРАТЬ ... ГДЕ Ссылка В (&СписокСсылок)";
Запрос.УстановитьПараметр("СписокСсылок", СписокСсылок);
Результат = Запрос.Выполнить(); // 1 запрос
```

### Правила написания запросов

1. **Проверить метаданные** через MCP `search_metadata` перед написанием запроса
2. **Текст запроса** — на новой строке, не в строке с `Запрос.Текст =`
3. **Результат** — всегда в промежуточную переменную; цепочки вызовов запрещены
4. **Псевдонимы** — обязательны для всех полей через `КАК`
5. **Параметры** — только через `УстановитьПараметр()`, конкатенация строк запрещена
6. **Временные таблицы** — префикс `ВТ_`
7. **Соединения** — предпочитать `ВНУТРЕННЕЕ СОЕДИНЕНИЕ` над `ЛЕВОЕ СОЕДИНЕНИЕ`
8. **Измерения регистров** — фильтровать первыми
9. **`ПЕРВЫЕ N`** — использовать для ограниченных выборок
10. **Индексы** — все поля фильтрации и соединения должны быть проиндексированы

```bsl
// ПРАВИЛЬНО — явные псевдонимы, промежуточная переменная
Запрос.Текст = "
    |ВЫБРАТЬ
    |   Товары.Ссылка КАК Товар,
    |   Товары.Наименование КАК НаименованиеТовара,
    |   Цены.Цена КАК ТекущаяЦена
    |ИЗ
    |   Справочник.Номенклатура КАК Товары
    |   ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.ЦеныНоменклатуры КАК Цены
    |   ПО Товары.Ссылка = Цены.Номенклатура";

РезультатЗапроса = Запрос.Выполнить();
Выборка = РезультатЗапроса.Выбрать();

// НЕПРАВИЛЬНО — цепочка вызовов
Выборка = Запрос.Выполнить().Выбрать(); // Запрещено!
```

### Кеширование в циклах

```bsl
// Кеширование через Соответствие
КешКонтрагентов = Новый Соответствие;

Для Каждого Строка Из ТаблицаДанных Цикл

    ДанныеКонтрагента = КешКонтрагентов.Получить(Строка.Контрагент);

    Если ДанныеКонтрагента = Неопределено Тогда
        ДанныеКонтрагента = ПолучитьДанныеКонтрагента(Строка.Контрагент);
        КешКонтрагентов.Вставить(Строка.Контрагент, ДанныеКонтрагента);
    КонецЕсли;

    // Использование ДанныеКонтрагента

КонецЦикла;
```

### Оптимизация запросов

#### Временные таблицы вместо подзапросов

```bsl
// НЕПРАВИЛЬНО — подзапрос в соединении
Запрос.Текст = "
    |ВЫБРАТЬ ...
    |ЛЕВОЕ СОЕДИНЕНИЕ (ВЫБРАТЬ ... ИЗ ...) КАК ВложенныйЗапрос
    |...";

// ПРАВИЛЬНО — временная таблица
Запрос.Текст = "
    |ВЫБРАТЬ ...
    |ПОМЕСТИТЬ ВременнаяТаблица
    |ИЗ ...
    |;
    |ВЫБРАТЬ ...
    |ЛЕВОЕ СОЕДИНЕНИЕ ВременнаяТаблица
    |...";
```

#### Параметры виртуальных таблиц

```bsl
// НЕПРАВИЛЬНО — фильтрация в WHERE
Запрос.Текст = "
    |ВЫБРАТЬ * ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки()
    |ГДЕ Склад = &Склад";

// ПРАВИЛЬНО — фильтрация в параметрах виртуальной таблицы
Запрос.Текст = "
    |ВЫБРАТЬ * ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(, Склад = &Склад)";
```

#### Ограничение выборки

```bsl
// Всегда использовать ПЕРВЫЕ N для больших таблиц
Запрос.Текст = "ВЫБРАТЬ ПЕРВЫЕ 1000 * ИЗ Справочник.Номенклатура";
```

#### Составные типы

```bsl
// НЕПРАВИЛЬНО — запросы ко всем возможным типам
Запрос.Текст = "ВЫБРАТЬ Документ.Контрагент.Наименование ...";

// ПРАВИЛЬНО — указание конкретного типа
Запрос.Текст = "
    |ВЫБРАТЬ
    |   ВЫРАЗИТЬ(Документ.Контрагент КАК Справочник.Контрагенты).Наименование
    |...";
```

#### Функция ПРЕДСТАВЛЕНИЕ

```bsl
// Использовать для отображения без соединений
Запрос.Текст = "
    |ВЫБРАТЬ
    |   ПРЕДСТАВЛЕНИЕ(Товар) КАК ТоварПредставление,
    |   ПРЕДСТАВЛЕНИЕ(Склад) КАК СкладПредставление
    |...";
```

#### Множественные OR

```bsl
// НЕПРАВИЛЬНО — индекс не используется
|ГДЕ Статус = &Статус1 ИЛИ Статус = &Статус2

// ПРАВИЛЬНО — использовать В ()
|ГДЕ Статус В (&СписокСтатусов)
```

## Запрещённые практики

| Практика | Причина | Альтернатива |
|----------|---------|--------------|
| `Сообщить()` | Устаревший метод | `ОбщегоНазначения.СообщитьПользователю()` (сервер) / `ОбщегоНазначенияКлиент.СообщитьПользователю()` (клиент) |
| `Выполнить()` / `Вычислить()` | Безопасность | Явная логика |
| `?(Условие, Знач1, Знач2)` | Читаемость, отладка | `Если...Тогда...Иначе` или вспомогательная функция |
| Венгерская нотация | Избыточность | Бизнес-имена |
| Глобальные имена как переменные | Конфликты | Другие имена |
| Запросы в циклах | Производительность | Пакетные запросы |
| Пустые исключения | Скрытие ошибок | Логирование + обработка |
| `ЗаписьЖурналаРегистрации()` | Шум в журнале | Только когда явно требуется задачей |
| Сравнение с `Истина`/`Ложь` | Избыточность | Использовать булево выражение напрямую |
| Yoda-синтаксис `Если 0 = Сумма` | Читаемость | `Если Сумма = 0` |
| Магические числа | Непонятность | Именованные переменные/константы |
| Хардкод учётных данных | Безопасность | БСП "Безопасное хранение паролей" или защищённые константы |
| `ТекущаяДата()` на сервере | Рассинхронизация | `ТекущаяДатаСеанса()` |
| Псевдообласти через комментарии | Нестандартно | `#Область` / `#КонецОбласти` |
| Параметр функции как доп. выход | Запутанность | Возврат через возвращаемое значение |
| Цепочки вызовов методов результата | Отладка | Промежуточная переменная |
| Смешивание Асинх с синхронным | Ошибки | Единообразный подход |

### Условия

- Сложные условия (3+ конструкций) — **выносить в отдельный метод**
- Добавлять пустые строки перед и после тела условий/циклов для читаемости

### Параметры функций

Параметр функции **НЕ ДОЛЖЕН** использоваться как дополнительный выход. Для дополнительных параметров — паттерн функции-конструктора:

```bsl
Функция ПараметрыЗаполнения() Экспорт
    Параметры = Новый Структура;
    Параметры.Вставить("Дата");
    Параметры.Вставить("Валюта");
    Параметры.Вставить("ПересчитыватьСумму", Истина);
    Возврат Параметры;
КонецФункции
```

## Комментарии модификации типового кода

Маркеры используются **только при изменении типового (стандартного) кода** в модулях базовой конфигурации.

### Формат

- Открывающий: значение `{COMMENT_OPEN}` из `.dev.env`
- Закрывающий: значение `{COMMENT_CLOSE}` из `.dev.env`
- Пробел после `//` обязателен

### Изменение типового кода

Удалённый типовой код — **закомментировать, НЕ удалять:**

```bsl
// {COMMENT_OPEN}
НовоеЗначение = {PREFIX}ПреобразоватьЗначение(Значение1);
// ТиповаяПроцедура(Значение1, Значение2);
ТиповаяПроцедура(НовоеЗначение, Значение2);
// {COMMENT_CLOSE}
```

### Новые процедуры в типовых модулях

Комментарий **внутри** процедуры, после заголовка:

```bsl
Функция НоваяФункция(Параметр) Экспорт
    // {COMMENT_OPEN}
    // ... код ...
    Возврат Результат;
    // {COMMENT_CLOSE}
КонецФункции
```

### Полностью новые (нетиповые) объекты

В модулях новых объектов (с `{PREFIX}`) — маркеры на каждый метод **НЕ НУЖНЫ**. Вместо этого — **один блок в заголовке модуля** с описанием объекта.

## Расширения конфигурации

### Приоритет способов модификации

```
Подписки на события > Расширения > Модификация типового кода
```

### Директивы расширения

- Предпочтительно: `&Перед` и `&После`
- `&Вместо` — только для функций, обязательно с `ПродолжитьВызов()`

### Правила размещения

| Что | Где |
|-----|-----|
| Новые объекты метаданных | Основная конфигурация (если `{NEW_OBJECTS_IN}` = `main_configuration`) |
| Реквизиты типовых объектов | Основная конфигурация |
| Типовые роли | **НЕ модифицировать** — создавать новые с `{PREFIX}` |
| Маршрутизация подписок | Общий модуль `{PREFIX}ПодпискиНаСобытия` |

### Формы в расширениях

Визуальное редактирование форм в расширениях **минимизировать** — реализовывать изменения программно через код.

## Платформенные правила

### Проверка пустых значений

Использовать `ЗначениеЗаполнено()` вместо прямых сравнений — корректно работает с пустыми ссылками и неинициализированными составными типами.

### Длительные операции

Операции свыше 10 секунд — выносить в фоновые задания с индикацией прогресса. Использовать БСП `ДлительныеОперации.ВыполнитьФункцию()`.

### Дата и время

На сервере: `ТекущаяДатаСеанса()` вместо `ТекущаяДата()`.

### Транзакции

- Учитывать наличие внешней транзакции (например, в обработчиках объектов)
- Не открывать вложенные транзакции без необходимости
- Транзакции — короткие, без интерактивных действий внутри

### Безопасность

- `УстановитьПривилегированныйРежим()` — только когда необходимо, всегда парно с восстановлением
- Учётные данные — через БСП "Безопасное хранение паролей" или защищённые константы
- Учитывать требования RLS при проектировании

### Кросс-платформенность

- COM-объекты запрещены (если явно не требуются задачей)
- Разделитель пути: `/` или платформенные функции, не хардкод `\`

### Проверка версии платформы

Перед использованием метода платформы — убедиться, что он существует в `{PLATFORM_VERSION}`.

## Code Smells — запахи кода

| Запах | Проблема | Решение |
|-------|----------|---------|
| **Data Clumps** | Одни и те же 3+ параметра передаются повторно | Объединить в Структуру |
| **Primitive Obsession** | Строки вместо перечислений, коды вместо ссылок | Использовать `Перечисление`, `СправочникСсылка`, `ОпределяемыйТип` |
| **Divergent Change** | Один модуль меняется постоянно по разным причинам | Разделить по ответственности |
| **Shotgun Surgery** | Одно изменение требует правок в 5+ местах | Консолидировать логику в общий модуль |
| **Feature Envy** | Метод интенсивно использует данные другого объекта | Переместить в общий модуль того объекта |
| **Variable Reuse** | Одна переменная хранит разные значения на разных этапах | Отдельная переменная для каждого значения |

## Внутреннее ревью после каждого изменения

После любого изменения кода — краткое саморевью:
- Стиль и читаемость
- Корректность и граничные случаи
- Безопасность
- Конкурентность / блокировки / транзакции
- Соответствие BSL Language Server

При обнаружении проблем — исправить и повторить цикл.
